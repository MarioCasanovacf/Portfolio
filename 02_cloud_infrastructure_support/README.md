# 02 — Cloud Infrastructure Support Operations Analytics

**Domain:** Enterprise support operations for cloud/HCI infrastructure
**Status:** Complete (5 notebooks + synthetic data engine)
**Approach:** Data sovereignty — full synthetic dataset generated from statistical models of real operations

---

## Business Context

Large cloud infrastructure companies manage thousands of enterprise customer environments simultaneously. Each cluster running in a customer's data center generates telemetry (IO latency, CPU, IOPS) and can produce support tickets when something goes wrong. The support organization's job is to resolve issues fast, prevent escalations, and maintain SLA compliance across global regions.

This is a domain where the cost of a missed signal is measured in enterprise contract renewals. The analytics framework here demonstrates how to move support operations from reactive (fix what breaks) to proactive (catch it before it breaks).

---

## The Four-Layer Framework

```
Layer 1 — Descriptive   →  What is happening?        (KPIs, SLA heatmaps, NPS trend)
Layer 2 — Diagnostic    →  Why is it happening?      (STL decomposition, GESD anomalies)
Layer 3 — Predictive    →  What will happen?         (SARIMA forecasting, staffing model)
Layer 4 — Prescriptive  →  What should we do?        (Escalation risk classifier)
Layer 5 — Engineering   →  How does this go to prod? (REST API integration pattern)
```

## Data

All data is synthetic — generated by `src/data_generator.py` to match the statistical properties of real enterprise support operations:

- **100,000 support tickets** with realistic TTR distributions (log-normal), priority mix, regional patterns, migration impact, escalation probability engine, and NPS inverse correlation with SLA breach
- **54,750 telemetry records** from 50 clusters × 3 years (daily grain) with injected anomaly days and weekday seasonality
- **24 migration cohort records** (VMware-to-hypervisor transitions)


## Statistical Techniques Reference

| Technique | Why chosen | Alternative considered |
|---|---|---|
| Log-normal TTR | Support TTR is zero-bounded with heavy right tail | Normal: fails for skewed data |
| Additive STL decomposition | IO latency spikes are absolute, not proportional to trend | Multiplicative: incorrect for fixed-magnitude anomalies |
| GESD over simple z-score | Handles masking effect from multiple simultaneous outliers | Rolling z-score: misses clustered anomalies |
| SARIMA(1,1,1)(1,0,1)[4] | Captures weekly + monthly seasonality in ticket volume | Prophet: higher accuracy but less interpretable |
| Random Forest over single model | Captures non-linear feature interactions (tier × migration × SLA) | XGBoost: marginally better AUC, less interpretable |
| Bootstrap CI for NPS | NPS is integer-distributed, not normal | Gaussian CI: produces incorrect interval widths |
