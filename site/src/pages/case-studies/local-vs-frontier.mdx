<Prose>

When building applications that process thousands of text inputs across multiple languages, the default choice is to make an API call to a commercial model. It is convenient, fast to prototype, and relies on the reasoning capacity of frontier models like OpenAI's GPT-5.5 or Anthropic's Claude Opus 4.8. Yet, deploying this architecture to high-volume production pipelines introduces significant trade-offs: high recurring token costs, network-dependent latencies, lack of data residency guarantees, and the risk of silent model degradation under the hood.

This case study documents a benchmark of the alternative pattern: using a large frontier model during the development cycle to bootstrap, label, and audit datasets, while training and deploying a lightweight, local model for production inference. The frontier model is the tool I build the solution *with*, alongside the engineer; the local model is the artifact I *deploy*.

To evaluate this pattern, I fine-tuned a multilingual classifier locally and measured its accuracy, F1 score, determinism, and inference throughput, comparing it against the July 2026 performance and pricing benchmarks of leading frontier APIs.

</Prose>

<Section n="01" slug="the-pre-registered-gate" title="The pre-registered gate: avoiding self-graded success.">

<Prose>

To prevent the common trap of selecting metrics after seeing model outputs, I fixed the evaluation rules beforehand. First, all metrics must be reported against a completely held-out test split of the Multilingual Amazon Reviews Corpus (MARC) consisting of 500 reviews per language across six languages: English, German, Spanish, French, Japanese, and Chinese. Second, star ratings (1–5) are mapped to three sentiment classes: Negative (1–2 stars), Neutral (3 stars), and Positive (4–5 stars).

Finally, I established a hard baseline gate: the local model's F1 score must stay within five percentage points of published classification benchmarks on this dataset, or it is deemed a failure. Additionally, the development cost must justify itself; if the break-even point against modern commodity APIs exceeds standard operational horizons, the local model represents a commercial loss.

</Prose>

</Section>

<Section n="02" slug="local-training-mps" title="Local training: fine-tuning on Apple Silicon.">

<Prose>

My local model is a fine-tuned `distilbert-base-multilingual-cased` (135 million parameters). I chose a distilled model to optimize for speed and memory footprint, making local execution on standard hardware practical. The training dataset consists of a stratified sample of 2,000 reviews per language (12,000 total), fine-tuned for two epochs.

Training was performed locally using PyTorch's Metal Performance Shaders (MPS) backend on an Apple M5 chip with 24 GB of RAM, completing in 516 seconds (under nine minutes) with a marginal training cost of zero. On the frozen test set, the model achieved an overall accuracy of 68.63% and a macro-F1 of 60.94%. Crucially, the model passed the determinism check, producing identical prediction logits across successive evaluations.

</Prose>

<Section n="03" slug="the-2026-frontier" title="The 2026 frontier: pricing and throughput.">

<Prose>

To build the cost and performance curves, I collected verified API pricing and speed metrics from July 2026. The cost model assumes an average query length of 100 input tokens (system instructions + review text) and 10 output tokens (structured sentiment predictions). Under this workload, the per-million-classification costs map to:
- GPT-5.5: \$5.00 input, \$30.00 output (\$0.000800 per review).
- Claude Opus 4.8: \$5.00 input, \$25.00 output (\$0.000750 per review).
- Gemini 3.1 Pro: \$2.00 input, \$12.00 output (\$0.000320 per review).
- DeepSeek V4 Flash: \$0.14 input, \$0.28 output (\$0.000017 per review).
- Local Model: \$0.00 marginal cost.

The throughput numbers reveal structural differences. Queried sequentially, network overhead limits throughput to 3 to 5 reviews/second. However, if the workload is parallelized, the throughput is capped by the provider's token-per-minute (TPM) and request-per-minute (RPM) limits. Under high-tier caps, the concurrent API throughput ceilings scale to 166.67 reviews/second for GPT-5.5, 120.0 reviews/second for Gemini 3.1 Pro, and 32.83 reviews/second for Claude Opus 4.8.

In comparison, the local DistilBERT model yields **237.97 reviews/second** on the MPS local GPU and 113.91 reviews/second on CPU. While high concurrency narrows the throughput gap, and even allows a concurrent API like GPT-5.5 to exceed the local CPU, the local model's main advantage shifts to local execution speed (~4 ms per review, without network transit), zero risk of hitting rate limit throttling, and complete independence from external service availability.

</Prose>

<Section n="04" slug="the-break-even-frontier" title="The break-even frontier: cost modeling.">

<Prose>

The calculation of the economic break-even point is governed by how engineering time is accounted for. If a business pays for development as a marginal freelancer invoice (modeled at \$800 for setup), the cost break-even point against DeepSeek V4 Flash lies at roughly 47.6 million classifications, meaning model ownership is only cost-effective at large scale.

However, in an in-house environment where engineer salaries are already fixed operational costs, the marginal setup cost consists only of the developer's tools (such as a \$20/month Claude subscription). In this in-house scenario, the economic break-even point against DeepSeek V4 Flash drops to 1.2 million reviews, and against high-tier APIs (like Claude Opus 4.8 or GPT-5.5) it is reached in just 26,000 to 27,000 reviews. For an in-house team, the local model pays for itself almost immediately.

In this context, the local model is not a niche option; it is a practical default for stable, high-volume classification. It processes reviews with local latency (~4 ms per review, bypassing network transit) and absolute data privacy.

</Prose>

</Section>

<Section n="05" slug="model-ownership-sustainability" title="Model Ownership and Sustainability.">

<Prose>

I limit this case study to well-defined, high-volume classification tasks. For multi-step reasoning, unstructured generation, or open-ended dialogue, frontier models remain the appropriate tool. This aligns with the core pattern: renting the frontier for development and dataset construction, but deploying an owned local model for stable inference operations.

Beyond the monetary break-evens, model ownership offers structural benefits. Running a 135-million parameter DistilBERT model on local hardware consumes orders of magnitude less energy per inference than sending queries to a hundred-billion parameter frontier model hosted in a remote data center.

To determine if this pattern fits your workload, the first step is to calculate your monthly classification volume and run the break-even math in this notebook with your own operational figures. For stable pipelines, transitioning from rented APIs to owned local models remains the most sustainable engineering path.

</Prose>
