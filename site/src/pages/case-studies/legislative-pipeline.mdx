<Section n="01" slug="data-a-server-will-render" title="Where the record actually lives.">

<Prose>

There is no API. The Chamber of Deputies publishes every recorded vote of every
deputy, but it publishes them the way a 2008 intranet would: one PHP page per
party group per vote, surnames first and no comma, the party named in a header and
nowhere in a field. The academic world has done better for the older
legislatures (Eric Magar's *dipMex* corpus ships legislatures 60 and 61 as a
clean roll-call *matrix*, every cell a vote code keyed to a stable legislator id),
but it stops around 2015. The legislature anyone actually wants to study, the
2024–2027 supermajority, exists only as that intranet.

So the pipeline has two capture paths. dipMex is a static download: the matrix is
unpivoted to one row per legislator per vote and lands as-is. SITL is scraped, and
SITL fights back. It blocks non-browser user agents; it paginates a 500-seat
chamber into party-group pages addressed by a parameter, `partidot`, that *looks*
like a party id and is not.

</Prose>

<Callout kind="aside">

`partidot` is a per-vote page index, not a stable party id: the same value maps to
different parties in different votes, and a single party can spill across pages.
The first capture trusted it, plus a "stop at the first empty page" shortcut, and
silently collected one party of \~37 deputies per vote instead of the whole floor
of \~500. The fix is to read the party from each page's header and to enumerate the
pages from the per-vote `estadistico` summary rather than guessing the range. The
lesson is the usual one: an HTML table will let you be wrong without telling you.

</Callout>

<Prose>

The check that the capture is *right*, not merely plausible, is that SITL prints an
official tally for each vote (so many for, against, abstaining, absent, per party).
Those tallies are never read into the fact table; they are held back as an
adversary. A dbt test aggregates our independently scraped per-deputy votes and
asserts the counts equal SITL's official totals. They match on all 274 votes of
the current legislature, every column.

</Prose>

</Section>

<Section n="02" slug="one-dimension-two-times" title="When did a vote happen, and when did it count?">

<Prose>

A deputy's party is not a fixed fact. It is a fact *as of a date*. Deputies change
caucus mid-term, so if you store party as a single column, every vote a deputy cast
*before* the switch is retroactively credited to the party they *ended* in. The
record quietly rewrites its own past.

So `dim_legislator` stops storing party as one value and stores it as a sequence of
dated rows: *this deputy belonged to MORENA from September to November, then to the
PT from November on.* Each row is one validity span. `fact_vote` then looks a vote
up by date: it picks the span whose half-open interval
`[effective_from, effective_to)` contains the day the vote was cast, so an October
vote resolves to MORENA and a December vote to the PT. (The pattern has a textbook
name, a slowly-changing dimension, and the half-open interval is what guarantees
every vote lands in exactly one span.)

The catch is that the two sources tell you about time differently, so a "span" has
to mean two different things. SITL stamps the party label on *every single vote*, so
the exact day a deputy's label changes is visible in the data, and a span can begin
on that day. dipMex can't do that: it gives a roster with nominal entry and exit dates,
and those dates turned out too loose to trust, with about 4% of real votes falling
*outside* the window the roster claims the deputy was even seated. So for the older
legislatures the only boundary solid enough to bound a vote is the legislature
term itself, one span per deputy per term.

</Prose>

<Prose>

Building SITL's spans is a classic *gaps-and-islands* problem. Walk each deputy's
votes in date order; every time the party label differs from the vote before it,
start a new "island"; collapse each island into one dated span. The result is a
single dimension carrying two kinds of span (a whole term for the old source, a
dated party-run for the new) under one key built so the two id-spaces can never
collide.

</Prose>

<CodeBlock lang="sql" file="dim_legislator.sql" width="wide">

```sql
-- SITL branch: a new version each time the party label changes between votes.
WITH flagged AS (
    SELECT legislator_id, vote_date, party,
           CASE WHEN party IS DISTINCT FROM
                LAG(party) OVER (PARTITION BY legislator_id ORDER BY vote_date)
                THEN 1 ELSE 0 END AS is_change
    FROM sitl_daily                      -- one party per deputy per day
),
islands AS (
    SELECT *, SUM(is_change) OVER (PARTITION BY legislator_id ORDER BY vote_date) AS island
    FROM flagged
),
runs AS (
    SELECT legislator_id, MIN(party) AS party, MIN(vote_date) AS effective_from
    FROM islands GROUP BY legislator_id, island
)
SELECT legislator_id, party, effective_from,
       COALESCE(
           LEAD(effective_from) OVER (PARTITION BY legislator_id ORDER BY effective_from),
           term_end + 1
       ) AS effective_to                 -- half-open: a vote on the boundary is the next run's
FROM runs;
```

</CodeBlock>

<Prose>

The fact table then unions both sources and resolves each vote against the
dimension on `(source, legislator_id)` and the date interval. The result is
798,445 attributed votes across three legislatures (296,200 and 365,298 from
dipMex, 136,947 from SITL), with not one orphaned vote. The same surrogate key,
`md5(source · legislator_id · legislature · effective_from)`, is a function of
business dates only, so it never churns between runs and the fact's foreign key
stays valid.

</Prose>

</Section>

<Section n="03" slug="what-the-count-says" title="What the count actually says.">

<Prose>

With the votes attributed, the obvious first question is party discipline. The
Rice index measures it per party per vote (the margin between a caucus's *for* and
*against* counts, normalised) and runs from 0 (a caucus split down the middle) to
1 (a caucus voting as one).

</Prose>

<Equation n="3.1" width="inline">

$$
\mathrm{Rice}_p(v) \;=\; \frac{\bigl|\,\mathrm{for}_p(v) - \mathrm{against}_p(v)\,\bigr|}{\mathrm{for}_p(v) + \mathrm{against}_p(v)}
$$

</Equation>

<Prose>

Averaged over the legislature's 274 votes, the answer is the same for everyone:
every caucus, government and opposition alike, votes as a near-perfect bloc. No
party falls below 0.99. This is not a chamber where party lines get negotiated on
the floor, because a caucus already this unanimous leaves a floor speech nothing to
move, no wavering member left within reach of the vote that would show it; it is
one where those lines are settled beforehand, because unanimity this total is
exactly what a decision reached in caucus and merely ratified in session looks like
once you count it. Same fact, read from both ends: nothing happens on the floor
because everything already happened off it.

</Prose>

<Prose>

Discipline that total invites a second question — who, then, ever switched sides?
The dated dimension answers it directly: 82 deputies hold more than one party
version. Reported as a headline, that is a story about a volatile congress. It is
also wrong.

Of the 82, only 15 are *monotonic* switches, a clean, one-directional move
through distinct parties, the shape an actual defection makes. Fifteen. The other
67 are *oscillation*: a deputy whose label flips back and forth between the same two
or three parties, vote after vote. Those parties are almost always MORENA, PVEM,
and the PT (the three members of the governing coalition), and
the flipping is SITL labelling the same coalition deputy inconsistently from one
vote to the next, not anyone crossing the floor. The dimension records the label
faithfully; the classification is what keeps the record from lying.

</Prose>

<Section n="04" slug="two-blocs-not-six" title="Two blocs, not six.">

<Prose>

If every caucus votes as a bloc, two questions follow. Is that level of discipline
normal, or particular to this chamber? And do the six blocs themselves cluster
into something larger?

The older source answers the first. In the 60th and 61st legislatures (2006–2012)
discipline was real but uneven: the tightest caucuses sat near 0.98–0.99, yet
opposition groups ran visibly looser (the PRD around 0.92, the PT as low as 0.87).
The current chamber has no such tail. Its *least* cohesive caucus, the PT at 0.993,
is more unified than the *most* cohesive caucus of either earlier legislature. Party
discipline is not new in Mexico; the uniformity of it is.

</Prose>

<Prose>

The second question needs a different measure. The Rice index scores discipline
*within* a party; to ask who votes *with* whom, reduce each party on each vote to
the side its present majority took, then for every pair of parties count the share
of votes where the two sides agree. Two parties are an effective bloc when that
share approaches one and independent when it approaches one-half.

The six caucuses collapse into two. MORENA, the PVEM and the PT (the governing
coalition) agree with one another between 98.9% and 100% of the time: less an
alliance than a single body voting under three names. Facing them, the PRI, the PAN
and MC form a looser but unmistakable opposition, agreeing 88–94%. Across the aisle,
agreement falls to 58–70%. The PRI, to take one case, votes with the PAN 94% of the
time and with MORENA 58%; that gap alone identifies it as an opposition party,
without anyone telling the model which side it sits on.

</Prose>

<Section n="05" slug="what-it-does-not-claim" title="What the warehouse does not claim.">

<Prose>

The pipeline attributes votes; it does not adjudicate motives. A few boundaries are
worth stating plainly.

</Prose>

<Callout kind="limits">

- No cross-era identity. dipMex keys deputies by a *seat* id, SITL by its own
  per-legislature `iddipt`. The same person across the 2009 and 2024 chambers is
  two rows, deliberately: linking them needs a fuzzy name bridge, which is built
  but reserved, since the conformed dimension is already correct at the (source,
  legislator, span) grain without it.
- A switch is a label run, not a verdict. `monotonic_switch` is the shape of a
  defection, not proof of one; intra-coalition reassignment produces the same
  shape. The view exposes the structure and leaves the reading to the analyst.
- Coverage is honest, not total. dipMex gives complete roll-calls only for
  legislatures 60–61; SITL covers the current 66 fully, with 64 and 65 rostered but
  not yet scraped. Cohesion and alignment here are descriptive — a measurement, not
  a cause.

</Callout>

<Prose>

What the warehouse does claim, it earns. A vote that lived only as a row in an
HTML table becomes a dated fact, attributed to whoever held the seat the day it was
cast, reconciled to the chamber's own official totals on every one of 274 votes.
The discipline of the data turns out to mirror the discipline of the chamber: read
it carefully and the dramatic number — 82 — settles into the quiet, true one.

</Prose>

</Section>
