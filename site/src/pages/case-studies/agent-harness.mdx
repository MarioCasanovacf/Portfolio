<Prose>

An orchestration layer for AI agents earns its keep only if it prevents a failure a
competent operator would otherwise hit, and does so mechanically. The shared board you
are reading about coordinated the construction of this portfolio, but "it built the site"
is an uninteresting claim. The question is whether the coordination mechanism did the work,
and how it performs under the realities of everyday operations.

This substrate is not an agent framework. It provides no agents, no prompts, and no memory
buffers. It is a shared task board, file locks, and a hand-off protocol. Nine small Python CLIs
manage the workspace, enforcing rules by refusing commands rather than by requesting compliance.
By separating the coordinator role from the worker sessions, it allows a group of cheaper,
parallelized models to execute coding tasks under the governance of a frontier planner.

</Prose>

<Section n="01" slug="the-era-after-fable-5" title="The Era after Fable 5: Coordinating a Society of Cheaper Models">

<Prose>

Building a codebase with autonomous agents requires addressing the cost of frontier intelligence.
Using a frontier model (such as Claude Fable 5) for every minor edit or file inspection is
prohibitively expensive. The harness solves this by establishing a society of agents, using
proportional verification to control resource consumption. The planner and the evolution
analyst reside on the frontier tier (Opus), while the daily workers and verifiers run on a
cheaper, faster tier (Sonnet/Gemini 3.5 Flash).

This division is codified in the bench registry (located in `.harness/state.json`). Workers lock,
edit, and test files using cheaper tokens, while high-stakes decisions (goal decomposition and
verification verdicts) override to the frontier model only per-dispatch. The routing rule is a
design choice: routine work is meant to run on the cheaper workers, with the expensive model
reserved for the synthesis of the task DAG. How much usage that actually shifts, and under what
conditions the rule holds, is still being measured — the honest figure is not yet in.

</Prose>

<Section n="02" slug="three-plane-substrate" title="The Three-Plane Substrate: Not a Classic CAR Agent">

<Prose>

In classic agent research, the Control-Agency-Runtime (CAR) decomposition is defined as
the internal architecture of a single agent. The harness wrapper surrounds a frozen LLM to
form a functional agent (yielding the formula llm + harness = agent).

By contrast, this Universal Agent Harness is a multi-agent coordination harness. It sits
outside the model boundaries, operating with zero LLM API calls in the coordination plane
unless configured otherwise. Rather than wrapping a single LLM to make an agent, it coordinates
multiple independent agent sessions (such as Claude Code sessions) working on the same repository.
It structures this coordination across three planes:

- **Control Plane**: The engine-agnostic contracts, task dependency DAG rules, and hard
  limits (defined in `ORCHESTRATION.md` and `state.json`). It uses no LLM APIs and governs
  the execution cycle.
- **Agency Plane**: The local execution utilities (nine stdlib Python CLIs like `blackboard.py`,
  `lock.py`, and `recontext.py`). Agents use these tools to interact with the workspace.
- **Runtime Plane**: The local workspace file system and git history (under `.harness/`).
  It serves as the physical substrate for state persistence.

Individual agent sessions (such as Claude Code) are guided by NLAH instructions (defined in `claude.md` or `gemini.md`) that map their local actions onto these planes, translating prompt-level Control into environment-level Agency, persisted on the local Runtime.

</Prose>

This decomposition isolates code edits from enforcement logic, but its success depends

<Section n="03" slug="auto-evolving" title="Auto-Evolving: The §5A Loop and Governance">

<Prose>

To allow the harness to adapt without breaking its core invariants, I implement the
§5A evolution loop. Operational frictions are captured as log inputs, which are analyzed
by a specialized evolution analyst (running on Opus). This analyst generates evidence-cited,
falsifiable proposals. If these proposals survive automated regression validation, they are
applied through a human gate, incrementing the harness generation and logging a git commit.

In practice, this loop has executed across four audited generations (as recorded in the
source repository state file), producing twenty-six accepted evolutions. It is important to
represent this history honestly: these generations are provenance from the source repository
(`serenissima-harness`). This portfolio monorepo migrated the board fresh at generation 4,
starting with a clean local log.

</Prose>

<Prose>

Whether an autonomous loop can safely execute migrations without human intervention is
doubtful, I'll recommend, in order to expand its autonomy, to start by adding a dry-run
validator for proposed schema modifications. That targets schema drift directly, runs in
seconds, and protects the parser from malformed proposals.

</Prose>

</Section>

<Prose>

Analyzing this graph demonstrates the structural advantages of a multi-agent coordination
substrate over both a single un-orchestrated agent and a classic CAR wrapper:

- **Un-orchestrated Swarm vs. Enforced Locks**: Running multiple raw agents without a
  coordination plane inevitably leads to file collisions and overwritten code. In our run,
  the file-lock rule (enforced by TTL write-locks) was active across all tasks. Because the
  coordinator partitioned the work cleanly, the agents rarely fought over the same files.
  The locks stood guard, guaranteeing that a collision was structurally impossible, our quiet
  locks here shall not be confused with a lack of utility, as they are the contract working
  as intended in order to keep track of the work.
- **Classic CAR Stagnation vs. DAG Rigidity**: A classic CAR wrapper ties a single LLM to a
  hardcoded state machine. If that model loops or crashes, the entire workflow freezes. In our
  harness, the rigidity of the task DAG and the lease sweeps solve this. In order to
  demonstrate this mechanical safety, please see the two failed tasks (T-113 and T-114): When
  the launch gate was tightened mid-flight to include more upstream tasks, the CLI refused to
  allow the coordinator to mutate in-flight dependencies or reuse IDs. Rather than allowing
  silent requirements drift (which would confuse a single agent or crash a serial pipeline) the
  harness forced the coordinator to abandon them as retired and explicitly declare the
  corrected paths as T-120 and T-121 as a tamper-proof system should.

<Section n="05" slug="operational-recommendations" title="Recommendations for Operational Usage">

<Prose>

Running a multi-agent coordination substrate successfully requires understanding its core
operational constraints. Rather than viewing task limits as friction, I recommend adopting
the following practices to prevent ad-hoc developer errors:

- **Respect Partitioning and Locks**: Always declare file write-locks via `lock.py` before
  modifying code. While the blackboard's task decomposition keeps concurrent agents working
  on disjoint files, locks remain your primary insurance policy against accidental overlaps.
- **Resolve Unknowns Up Front**: Seal all dependency edges and compile a plan using an
  up-front unknowns pass before committing the task DAG. Because the blackboard resists
  in-place edits to preserve execution stability, planning thoroughly prevents the need to
  retire and recreate tasks mid-run.
- **Audited Evolution Governance**: Introduce changes to the coordination CLIs and prompt
  structures exclusively through logs and audited proposals. Evolution history (such as the
  twenty-six mutations) should be maintained as a versioned, git-tracked biography.
- **Rotate Verifier Identities**: Rotate the agent identities assigned to validation roles.
  Enforcing cognitive diversity prevents verifier monoculture and ensures independent reviews.

</Prose>

<Callout kind="note">

- Partition workspace tasks cleanly to keep workers off each other's paths, letting the TTL
  write-locks act as automated insurance.
- Resolve dependencies up front before publishing the DAG; if requirements change, retire
  the old task IDs and declare the corrected paths as new tasks.
- Keep NLAH modifications and CLI updates inside the §5A evolution loop, versioning the
  coordination contract under human approval gates.
- Rotate reviewer sessions to ensure verifiers remain independent from producers across
  different epics.

</Callout>
