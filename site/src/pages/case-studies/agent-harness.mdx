<Prose>

A coordination layer for AI agents earns its keep only if it prevents a failure a
careful operator would otherwise hit, and prevents it mechanically, so that being
careful stops being the only thing standing between the operator and the mess. That is
the test I want to hold this one to. It coordinated a good part of the portfolio you are
reading, and it would be easy to let "it built the site" stand in for "it worked." Those
are different claims, and only the second one is interesting.

So I want to be exact about what this is. It is not an agent framework: it hands you no
agents, no prompts, no memory, no runtime. It is a substrate that sits under agents you
already have (here, several Claude Code sessions and sub-agents) and gives the group of
them one shared task board, file locks, and a hand-off protocol. No database, no external
API, nine small Python CLIs that depend on nothing but the standard library. The whole of
it is a set of rules, and the point of the design is that each rule is a command that
refuses rather than a request an agent has agreed to honour.

The real question this page tries to answer, in the register this portfolio prefers, is
whether that substrate actually prevents the two failures it was built against, what it
demonstrably coordinated on this repository, and — the part a pitch would bury — where the
mechanism added friction, where a task failed, and what the board in front of me simply
cannot tell you.

</Prose>

<Section n="01" slug="two-ways-to-run-a-crowd" title="Two ways to run a crowd of agents, both bad.">

<Prose>

Run more than one coding agent against one repository and two failure modes show up
almost immediately. The first is unmanaged parallelism: no shared index of what needs
doing, no ownership of who is doing it, no gate on what depends on what. The agents
duplicate each other, overwrite each other's edits, and nobody actually delegates — they
just collide in the same files and call it collaboration.

The tempting fix is worse in a quieter way. You chain the agents into one long line, A
hands to B hands to C, all the way to N. That trades chaos for fragility. Every hop loses
context, and one stalled link freezes the whole pipeline behind it — the serial-stagnation
problem, where a pipeline is only ever as live as its slowest agent. Neither the free-for-all
nor the assembly line is what you want, and the interesting design sits between them.

</Prose>

<Prose>

The resolution the harness proposes is a single idea, and stating it plainly is most of
the work: delegation is a property of the task graph, not of an org chart drawn in
advance. Work is a set of tasks with dependency edges, and an edge exists only where one
task literally consumes another's output. A cascade is then not a management style you
impose everywhere; it is a fact about two specific tasks, and it lives exactly there and
nowhere else. Everything not joined by such an edge runs in parallel. That is the whole
philosophy, and the rest of this page is about whether the mechanism holds it up.

</Prose>

</Section>

<Prose>

Two things in that picture are the argument. First, nine tasks carry no incoming edge:
the two hosting-and-domain research tasks, the two site-move tasks, the notebook-path fix,
the legislative diagnosis, the notebook execution, the Heston assessment, and the
real-estate audit. Nothing joins them, so nothing forces them to wait, and they formed the
frontier a set of agents could claim at once. Second, the cascades that do exist are all
real: the legislative fix cannot be verified before it is implemented, which cannot happen
before it is diagnosed; the launch join on the bottom lane genuinely consumes the output of
eight upstream tasks and refuses to be claimable until every one of them is done. The graph
has exactly the edges the work has, and no more.

</Prose>

<Section n="03" slug="rules-a-command-refuses-to-break" title="Rules a command refuses to break.">

<Prose>

The distinction I care about is between a rule an agent has agreed to follow and a rule the
tooling will not let it break. An agreement degrades the moment the agent is confused,
adversarial, or simply mid-hallucination; a refusal does not. Three rules matter here, and
each is backed by a command that returns an error rather than by a line of documentation.

The cascade gate is the simplest. Asking to claim a task whose dependencies are not all
done is refused, because the claim command reads the graph and checks. There is no polite
version where an eager agent starts early on a task whose inputs do not yet exist; the
command declines and points it at an unblocked task instead.

The producer-does-not-approve-its-own-work rule is the one I most wanted to be mechanical,
because it is the one an agent under pressure most wants to skip. A worker cannot mark its
own task done. It can only hand off; a different agent has to render the verdict. In the
harness this is not an honour code — the command that sets a task to `done` reads the
hand-off record and refuses if the acting agent is the same one that produced the work:

</Prose>

<CodeBlock lang="python" file="blackboard.py (producer ≠ approver guard)" width="wide">

```python
# --status done is refused unless (a) a handoff record exists — the task was
# actually reviewed by someone — AND (b) the acting agent differs from the
# handoff's producer of record. Authorship governs, not current status.
handoff = task.get("handoff")
handoff_producer = handoff.get("from") if handoff else None

if handoff_producer == acting_agent and not override_producer_check:
    sys.exit(
        "refused: {} produced this task; a different agent must verdict it. "
        "Producer != approver is mechanical, not cultural.".format(acting_agent)
    )
```

</CodeBlock>

<Prose>

The escape hatch exists (a flag, which the command logs and which requires a written
reason) because a rule with no audited override tends to get worked around in ways that
leave no trace, which is worse. The third rule is the one that keeps a crashed agent from
freezing the board: claims carry a lease and file locks carry a time-to-live, so an agent
that grabs a task or a file and then dies or loops has its hold expire and its work returned
to the pool. Nothing an agent holds is held forever, which is the mechanical answer to the
serial-stagnation problem from §01.

</Prose>

<Section n="04" slug="what-it-coordinated" title="What it coordinated, and the one time it sent work back.">

<Prose>

Here is the part a reader can check against the board file rather than take on my word. Of
the twenty-two tasks, eighteen reached done, two failed (§05 is about those), and two remain
open behind the launch gate that waits on a domain purchase only I can make. Every one of the
twelve producer tasks — the thinking and the implementation work — was verdicted by a
different agent than the one that produced it, which is the producer-does-not-approve rule
doing exactly what it is for. The six verification tasks were themselves closed by an agent
other than the one that created them. Across the whole board, no agent graded its own homework,
and that is not a claim about diligence: it is what the command would have refused to allow.

The most useful single event on the board is the one where the review actually bit. The
real-estate implementation was handed off, reviewed, and rejected — the load-bearing number in
that study, a cross-validation score that collapses when neighbourhoods are held out of the
training fold, had been reported from a single split with no uncertainty band. The task went
back to open, the worker re-ran the comparison across six seeds and thirty folds, and only then
did a different agent accept it. A harness where the producer signs off on its own work does not
catch that, because the incentive at sign-off is to be finished, not to be doubted. The value of
the rule is precisely the cases where it turns a green checkmark back into open work, and on this
board it did so once, visibly, with the reopening recorded in the task's own history.

</Prose>

<Callout kind="note">

The wider provenance the substrate ships with — two standalone tools it built end to end
(a Markdown table-of-contents generator and a cron-expression explainer), two method
tournaments that promoted a byte-identical winner, and four generations of audited
self-modification — lives in the source repository's logs, not on this portfolio board,
which was migrated in at generation 4 and started a fresh evolution loop. I mention it so
the record is complete, and I keep the numbers on this page to what the board in this repo
can be made to confess.

</Callout>

</Section>

<Section n="05" slug="where-it-cost-me-something" title="Where it cost me something, and what the board cannot tell you.">

<Prose>

The two failed tasks on the bottom lane are the most honest thing on the board, so I would
rather lead the limits with them than tuck them away. They are the first two launch-configuration
tasks, and they did not fail because the work was wrong. They failed because I tightened the
launch gate after they were created — the full-quality version needed to wait on more upstream
work — and the coordination CLI has no command to edit a task's dependency edges once the task
exists, and it refuses to reuse a task id. The only lifecycle-legal move left was to abandon the
two tasks as failed and recreate them, with the correct edges, under new ids. So the board carries
two red nodes that mean "superseded by a stricter version of themselves," not "broke." That is
real friction the mechanism imposed: the same rigidity that makes the dependency graph trustworthy
is what left me unable to amend it in place, and the honest residue is two failures that a casual
reader would misread. I would fix that with a dependency-editing command before I would defend the
red nodes as elegant.

There is a larger thing the board cannot show, and it is the one I most want to be plain about.
The rule that agents cannot collide in the same file is enforced by write-locks, and the locks were
genuinely exercised — the notebook-execution task locked all five notebooks before rewriting them
in place, the reorg task locked a config file before editing it. But this board was decomposed
cleanly enough that two agents rarely reached for the same file at the same instant, so I cannot
point to a moment on it where the lock refused a real second writer and prevented a real collision.
The mechanism was present and correct; the adversarial event that would prove it bites lives in the
substrate's own concurrency smoke test, not here. Reporting the lock as load-bearing on the strength
of this board alone would be claiming more than the board earns.

What survives all of that is narrower than a pitch and, I think, more worth having. On one real
repository, a file-based board with a mechanically enforced hand-off kept a crowd of agents from
grading their own work, gated the one launch task on the eight things it genuinely depended on, let
nine independent tasks run without waiting on each other, and turned one over-confident result back
into open work before it could ship. That is the mechanism doing the specific job it claims, on
evidence you can re-read in the board file. Whether it was worth the friction of the retired tasks,
and whether the locks earn their place on a board busy enough to make two agents fight over one file,
are the next two questions — and if I were to keep only one instrument for answering them, it would be
the same append-only log that made this page checkable in the first place, pointed this time at a run
built deliberately dense enough to make the agents collide.
